## 渲染与收口

- waiting 小球、streaming 正文、settled 完成态共用同一条 assistant 消息 item，不再切第二滚动主人
- `ChatStreamingRenderer.kt` 当前 active streaming 和 settled Markdown 都走同一套 soft-wrap block renderer；它是面向移动聊天的轻量 Markdown 子集渲染器，不是 Markwon / CommonMark / GFM 全量引擎。当前支持段落、标题、列表、引用、加粗、斜体、行内代码、Markdown 链接、裸 URL、emoji / 普通 Unicode 文本，标准表格降级为手机可读的项目行；streaming 打字机不会拆开 UTF-16 代理对、扩展汉字、常见 emoji 组合簇、肤色 emoji、ZWJ 组合和旗帜。不承诺完整代码块高亮、复杂嵌套列表、GFM 表格网格、数学公式、Mermaid、图片 Markdown、任意 HTML 或全量 Unicode 分词。旧 committed 物理行预切 / TextMeasurer 渲染链已移除，避免 streaming -> settled 收口时换渲染模型导致行高 / 行宽微动
- 2026-06-15 按用户视频里“加粗处一伸一缩、末尾一口气吐出”的问题继续收口：renderer 新增 streaming-aware inline 模式，活跃 streaming 段遇到 `**...`、`*...`、`` `... `` 这类未闭合标记时会先按目标样式显示正文、隐藏结构符号；如果当前吐字帧刚好停在待续的 `**`、`*` 或反引号，也不再把 raw marker 短暂露出来。纯 `**`、`*`、反引号这类不可见结构标记不会单独替换 waiting 小球，等第一个可见字一起进入正文；streaming 期间最新视觉尾块始终按 streaming inline 规则处理，已换行 / 空行分隔出去的旧块如果仍含未闭合加粗、斜体或行内代码，也继续按 streaming inline 规则处理，避免在 streaming / settled 两套规则之间切换造成轻微伸缩。完成态仍要求闭合标记才吃掉符号；已离开尾部且 inline 标记完整闭合的旧块走 settled 模式和缓存。这样保持“边生成边渲染”，同时降低 `**` 闭合瞬间整行重排。DONE 收口不再取消 reveal job 并把剩余 buffer 一口气 flush，而是继续按当前打字节奏把 buffer drain 完再进入两阶段 finalize，避免后半段文字突然整段出现
- active Markdown 仍实时吐字，但 `# ` / `- ` / `1. ` / `> ` 这类结构前缀必须等后面已有非空正文才结构化，避免只有符号的半成品先变标题 / 列表再重排
- streaming reveal 当前按更接近正常聊天产品的节奏处理：远端首个 chunk 太快返回时，waiting 小球至少展示约 1.8 秒，保证现有 720ms 往返呼吸动画能被用户看见；中文通常 1 个字一拍，长英文 / 数字词块按不超过 8 个字符的块吐出，标点、省略号、破折号和换行有更明显停顿。该改动只调小球和吐字节奏，不改变正向 `LazyColumn` 滚动主链、工作线、今日农情列表项或失败 / 重试链路；仍不恢复新字尾部灰色高亮动画，也不把正文改成慢速朗读
- 标准 Markdown 表格当前不做真表格控件，renderer 会把表格行降级成普通项目行文本，保证模型偶发输出表格时至少可读、不撑乱聊天布局；代码块内的 `|` 不做表格降级。emoji / 表情若偶发输出，继续按普通文本由 Compose `Text` 承接；streaming reveal 只保证不拆半常见可见符号，不主动鼓励模型使用表情
- streaming 期间不提前显示免责声明文字；如果内容已满足免责声明触发条件，只预留同等几何高度，settled 后才显示真实文案，避免尾部收口当拍突然增高
- 完成态收口继续保留两阶段 finalize：
  - `beginPendingStreamingFinalize(...)`
  - fresh bounds 到位后 `finalizeStreamingStop(...)`
  - 不回退成同拍直接切 settled
  - 不在 finalize 当拍恢复旧 bottom align 精修；若仍处于 AutoFollow，只在 fresh bounds 到位后请求一次正向底部锚点，避免长回复完成时窗口被重新锚到上方
  - 极端切后台、布局回调丢失或 fresh bounds 长时间不到位时，最多等待约 1500ms 后也会退出 streaming 状态；超时路径不强制底部锚点，避免用旧几何把列表拉跳

- 异常与生命周期兜底当前继续围绕“不中断主滚动链”小范围加固：日额度耗尽后，当前本地会话当天会把发送键置灰但保留点击提示；额度耗尽时仍保留消息流里的 assistant 重试卡片，尾部文案复用统一的“回复未完成 · 点击重试”，今天点击只提示额度，第二天额度恢复后可沿用原重试链继续发同一条；限流、活动流进行中等明确不可立即重试的中断只保留对应中部提示，避免和输入框或其它黑色提示重叠。无网发送纯文字或带图片消息时都会生成失败用户气泡并同步清空输入框和草稿，图片失败气泡会保留本地缩略图和“发送失败 / 重发”入口，后续即使用户重新输入同文也会优先复用已有失败用户消息，不再刷出多条相同失败消息；输入区上方不再挂全局提示条，额度、网络、图片数量、输入过长、上传失败等短提示统一在主界面中部短暂浮层展示，并在菜单、会员页、图片面板或复制菜单打开时让位。文字 / 图片发送起点会同步保存本地聊天窗口，降低刚点发送就杀 App 时尾部用户消息或 assistant placeholder 来不及落盘的概率；远端历史模式下已删除本地固定示例假流式，冷启动只用真实后端 snapshot、本地 pending / 失败态和真实 streaming draft 恢复；如果用户在真模型吐字中极端切后台或系统杀进程，冷启动会把已保存的真实 streaming draft 恢复成同一用户消息下方的 `回复未完成 · 点击重试` assistant 失败气泡，并继续围绕当前 `clientMsgId` 拉取 `/api/session/snapshot`，服务端已完成则替换为完整 AI 回复，拉不到完整回复时也保留已吐出的真实半截内容而不是清成空白或伪装完成。Android 主聊天前台 SSE 和后台 `streamChatToCompletion` 都使用流式专用 OkHttp client，读取超时为 0，普通 API 仍保留 60 秒读超时；这样模型长回复、后台切换或网络短抖时不因为长时间无新 chunk 就被客户端误断。后端 streaming 进程被杀后若远端 snapshot 追不回答案，会在用户消息下方补一个 `回复未完成 · 点击重试` 的 assistant 失败入口；`/api/chat/stream` 打到后端后会由后端继续读完上游并在 DONE 后写入 `session_round_archive`，Android 切后台 / 锁屏再回来时会围绕当前 `clientMsgId` 持续拉取 `/api/session/snapshot`，若后端已完成则直接恢复完整 AI 回复，冷启动发现尾部用户消息缺少 AI 回复时也走同一恢复窗口；带图片发送当前已接入 WorkManager 延迟兜底：用户消息上屏并落盘后会按 `chatScopeId + userMessageId` 排一个唯一后台任务，前台活跃时后台只重试不抢跑，前台一旦开始 `/api/chat/stream` 会写入 10 分钟远端启动保护窗；如果 assistant 失败态已经有远端图片 URL，用户点重试时也会重新登记同一条 pending，让 App 立刻被杀后仍有后台补发机会；后端同步使用 `chat_stream_inflight` 进行中锁和 lease token，并通过数据库唯一约束限制同一 `user_id` 同时只有一条活跃主流式请求，避免不同 `client_msg_id` 并发打穿额度预检查并多开主模型。重复请求在完成归档前返回 `409 STREAM_IN_PROGRESS` 给客户端走长窗口 snapshot 恢复，完成后的 replay 真源以 `session_round_ledger` / 归档成功为准，服务端只在轮次归档成功后才向客户端发送 SSE `[DONE]`，额度扣减也在归档成功后执行，避免 quota ledger 已写、客户端已收完成态但回答未归档时返回空 replay；旧请求释放时也只能释放自己的锁。前台 SSE 会识别 replay 事件，不会把 replay + DONE 当成空回复完成。若前台上传失败并已经显示“发送失败”，对应后台任务会取消，不会一边显示失败一边后台偷偷消耗额度。切后台 / 锁屏时仍会收起输入焦点并清掉消息 / 输入选择菜单，避免回来还挂着复制黑卡片；`SessionApi` 的当前 SSE call 会等响应读循环退出后再清空引用，保证 reset / cancel 能尽量取消正在读的远端流

## 当前阶段判断

- mixed active-zone / overlay 架构经过多轮补丁后，已经证明不适合作为最终方案；继续补只会在“发送抖、拖动乱窜、消息重叠”和“闪烁 / 收口抖”之间来回换问题
- 反向列表单主人虽然压住了大部分底部冒头闪，但在最新 assistant item 内小幅上滑时仍会被动态长高锚点带一下，用户体感不可接受
- 当前仓库已按用户拍板切回“**单一运行时主人 + 正向列表**”路线；这不是恢复旧正向 overlay / placeholder / 追滚补偿，而是：
  - 正向 `LazyColumn`
  - 保留 send-start 保护窗口
  - 保留两阶段 finalize
  - 保留 unified soft-wrap renderer
  - 删除 mixed active-zone、小分割 itemization、streaming raw delta / scrollBy 追滚
