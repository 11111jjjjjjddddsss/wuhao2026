# Android 极端条件回归清单

本清单用于真机手测农技千问 Android 端在弱网、切后台、清数据、失败重试等极端条件下是否还能保持可用。它不是自动化测试脚本，也不是业务规则真源。

## 0. 测试前准备

- 优先用 release / profile 包看真实丝滑度；debug 包只用于文案预览和快速验证
- 每次只测一个变量：不要同时断网、切后台、发图、连点发送
- 出问题时记录：操作步骤、是否正在 streaming、键盘是否打开、是否在底部、屏幕录制

## 1. 网络相关

### 1.1 无网络时发送

步骤：
1. 打开 App
2. 关闭 Wi-Fi 和移动数据
3. 输入一句话并点击发送

期望：
- 不插入异常的 assistant 正在生成状态
- 输入框附近提示“当前网络不可用”
- 输入内容不应莫名丢失
- 连续点同一段文字的发送键时，应复用已有失败用户消息，不应刷出多条相同“发送失败”气泡

### 1.2 生成中断网

步骤：
1. 正常发送一条会生成较长回复的问题
2. streaming 过程中断网

期望：
- 已经生成的文字保留
- 回复尾部出现“回复未完成 · 点击重试”或输入框浮层出现“网络波动，回复未完成”
- 用户点击“重试”后能继续发起，不应重复插入多条乱消息
- 滚动链不应被失败态打乱，仍可上滑 / 回到底部
- 如果 App 被系统杀掉后回来，远端 snapshot 又追不回答案，应在原用户消息下方保留“回复未完成 · 点击重试”入口，而不是只剩用户问题

### 1.3 弱网 / 高延迟

步骤：
1. 用系统网络限速、热点弱信号或开发者工具模拟弱网
2. 发送长问题

期望：
- 小球可以等待，不应频繁闪烁或重复出现
- 若超时 / 中断，进入失败态而不是卡死
- 用户仍能滚动浏览已生成内容

### 1.4 服务忙 / 限流

触发方式：需要后端或模拟返回 rate_limit。

期望文案：
- “当前请求较多，请稍后重试”

### 1.5 今日额度耗尽

触发方式：需要后端或模拟返回 quota。

期望文案：
- “今日额度已用完，请明天再试”
- 同一 App 会话内，后端返回 quota 后，输入有文字时发送键应显示灰态；点发送 / 用户重发 / AI 重试都只弹该提示，不新插消息、不扣次数

## 2. 生命周期 / 切换场景

### 2.1 streaming 中切后台

步骤：
1. 发送长回复
2. streaming 过程中按 Home 或切到微信
3. 过几秒回到 App

期望：
- App 不崩溃
- 已生成内容不丢
- 如果流已中断，应显示失败 / 可重试，不应无限小球
- 回到页面后键盘状态不应把消息工作线顶乱
- 若切后台前有消息复制菜单或输入框选择菜单，回来后不应继续悬挂旧菜单

### 2.2 键盘打开时切后台

步骤：
1. 点输入框弹出键盘
2. 切后台再回来

期望：
- 输入框位置正常
- 消息列表不被键盘抬升
- 不出现旧输入框残影

### 2.3 锁屏 / 解锁

步骤：
1. App 前台，键盘打开或 streaming 中
2. 锁屏再解锁

期望：
- 不崩溃
- 已生成文字完整保留，不丢字、不截断、不只剩半段排版
- 如果流已中断，应显示失败 / 可重试，不应无限小球
- 消息区、输入框、键盘状态恢复合理

## 3. 滚动链 / 生成链

### 3.1 streaming 中上滑

期望：
- 用户上滑后不被强行抢回底部
- 底部仍可继续生成
- 回到底部按钮只在停止滑动后短暂出现

### 3.2 手动滑回底部

期望：
- 手指抬起且列表停在物理底部后，AutoFollow 应恢复
- 后续新字继续贴着 96dp 工作线吐出

### 3.3 完成态收口

期望：
- 回复结束后不大跳
- 工作线以下 96dp 空白露出
- 免责声明出现时不把尾部突然顶高

## 4. 输入框 / IME

### 4.1 冷启动首次点输入框

期望：
- 键盘弹起只移动输入框
- 消息列表不抬升
- release 包体感优先于 debug 包

### 4.2 发送后回缩

期望：
- 输入清空后输入框回到空态高度
- 键盘收起时不出现明显旧高度残影

### 4.3 超长输入

步骤：
1. 粘贴超过 6000 字
2. 点击发送

期望：
- 显示“已超过6000字，暂时不能发送”
- 不发送、不扣次数、不插入异常消息

## 5. 图片相关（图片入口完整上线前先作为底层能力清单）

### 5.1 非 JPEG / PNG

期望：
- “仅支持 JPEG / PNG 格式”

### 5.2 压缩后仍超过 1MB

期望：
- 当前文案较长，后续图片 UI 上线前建议再收口

### 5.3 上传服务未配置 / 响应异常

期望：
- 不扣次数
- 不进入主模型调用
- 给出可理解失败提示

### 5.4 超过 4 张图片

期望：
- 拦截并提示最多 4 张

### 5.5 只发图片

期望：
- 不再因为缺少文字描述而拦截
- 仍遵守最多 4 张、格式、大小和上传成功条件
- 模型应基于图片可见信息先给参考判断，并追问缺失的关键背景

## 6. 清数据 / 重装 / 换机

### 6.1 清除 App 数据后启动

期望：
- 当前无手机号 / 稳定账号登录时，应视为 clean-state
- 不应从本地恢复旧聊天、旧输入框高度、旧滚动位置
- 如果未来绑定手机号，后端按账号返回最近 30 轮属于业务恢复，不是 UI 回退

### 6.2 重装覆盖安装

期望：
- 本地状态不应把 UI 拉回旧样式
- 若未清数据，正常保留本机当前缓存 / user_id

## 7. 文案预览临时入口

当前 debug 包中，点击右上角菱形按钮可打开 UI 文案预览，点空白关闭。

要求：
- 只用于本轮文案核对
- 核对完成后应删除该 debug 面板
- 不进入正式 release 主界面

## 8. 当前最重要的人工验收顺序

1. release / profile 包下输入框弹起回缩
2. streaming 长回复上滑 / 下滑 / 回底
3. 生成中断网后的失败态和重试
4. 清数据后 clean-state
5. 文案预览面板逐条核对
