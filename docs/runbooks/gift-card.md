# 礼品卡 Runbook

最后更新：2026-06-16

## 当前状态

礼品卡首版后端、Android 用户兑换入口和管理后台已经接入。

当前真实能力：

- 后端新增礼品卡批次表、礼品卡表和兑换尝试表。
- 管理后台可创建 Plus / Pro 礼品卡批次，默认 30 天，可设置张数、天数和备注。
- 礼品卡首版按“生成后即可兑换、兑换成功立即生效”处理，不开放预约生效 / 未来生效配置。
- 后端单测已锁住后台批次创建默认 `valid_from=创建时间`、同账号重复兑换幂等成功、并发状态更新必须影响 1 行、Plus 兑换 Pro 生成升级补偿、Pro 兑换 Plus 被拒绝且不消耗礼品卡。
- 完整卡码会在创建批次成功响应返回，并使用 `APP_SECRET` 派生密钥 AES-GCM 加密保存到 `gift_cards.code_ciphertext`；数据库仍保留 HMAC-SHA256 hash、掩码和尾号用于兑换校验和追溯。
- 管理后台可查询批次、卡状态、兑换账号ID、脱敏手机号、兑换地区、兑换时间和会员到期时间。
- 管理后台可按批次、卡状态、账号ID和卡码尾号追溯礼品卡；新生成礼品卡可在后台列表直接查看完整卡码并复制，可按账号ID、卡码尾号、成功 / 失败和失败原因筛选兑换尝试。
- 管理后台 `GET /admin-api/v1/gift-cards/summary` 已提供全局卡状态汇总、最近 24 小时失败尝试数和最近 7 天失败原因聚合。
- `failure_reasons` 等列表在没有数据时按空数组处理；后台页面不应因为生产库 0 批次 / 0 卡而报错，监控面板会把“没有可兑换礼品卡”作为待处理事项提示。
- 监控面板里的礼品卡“可用”按当前可兑换口径统计：`status=active`、已到 `valid_from` 且未超过 `valid_until`；后台创建的新卡默认 `valid_from` 为创建时间，已过期的 active 卡不再计入“可用”数量。
- 管理后台可作废未兑换的 active 礼品卡，动作需要 `finance_ops` 或 `owner` 权限、二次确认、作废原因和审计。
- 用户侧接口 `POST /api/gift-cards/redeem` 已接入，必须登录后调用。
- Android 设置页“礼品卡”已经接到 `POST /api/gift-cards/redeem`，只在后端返回成功后展示成功结果并刷新会员权益；若后端标记 `replay=true`，App 展示“权益已生效 / 该礼品卡已兑换成功，会员权益已生效，无需重复兑换”。
- Android 失败提示按后端错误码收口：卡码错误提示“礼品卡码不正确，请核对后重试”，被别人兑换或作废提示“这张礼品卡已被兑换或已作废”。
- 礼品卡兑换会事务内发会员权益，并写兑换尝试和审计。
- 同一账号重复提交自己已经兑换成功的同一卡码时，用户侧接口按幂等成功返回既有兑换结果，响应带 `replay=true`，并额外记录一次成功尝试，避免 App 丢失首次 200 后再次输入同一卡码被误判为失败；其他账号重复兑换、已作废或已过期仍按失败处理。

当前仍未完成：

- 批量发放名单、发放对象管理和更细风控统计尚未开放。
- 本次加密保存能力上线前生成的旧卡没有 `code_ciphertext`，后台只能显示掩码 / 尾号，不能反推出完整卡码。
- 礼品卡不是支付订单，不替代正式支付、退款或对账系统。

## 兑换规则

- 礼品卡类型只支持 `plus` 和 `pro`。
- 默认权益周期为 30 天，后台创建批次时可调整，最大 366 天。
- Free / 已到期用户兑换 Plus / Pro：从当前时间开始计算到期时间。
- 同档用户兑换同档卡：从当前有效期向后顺延。
- Plus 用户兑换 Pro：升级为 Pro，并把 Plus 剩余权益折算为升级补偿次数。
- Pro 用户兑换 Plus：拒绝兑换，不消耗礼品卡。
- 已兑换、作废或已过期的卡不会再次发权益；同一账号重复提交自己已兑换成功的卡码只返回既有成功结果，并在 App 里提示“权益已生效”。

## 数据真相

- `gift_card_batches`：批次名、档位、天数、数量、有效期、创建人、备注。
- `gift_cards`：卡 hash、掩码、尾号、档位、状态、兑换账号ID、兑换时间、地区、会员到期时间。
- `gift_card_redemption_attempts`：兑换尝试、成功 / 失败、失败原因、脱敏 IP、地区和时间。
- `user_entitlement`：会员当前档位和到期时间。
- `upgrade_credits`：Plus 升 Pro 时的补偿次数。
- `admin_audit_logs`：后台生成 / 查询和用户兑换相关审计。

不要把完整卡码明文写进备注、作废原因、审计 detail、日志、文档或聊天记录。当前生产设计允许后台登录角色在礼品卡列表查看完整卡码，但完整卡码必须以加密字段保存，兑换仍使用 HMAC hash 校验。

## 接口

后台：

- `GET /admin-api/v1/gift-cards/batches`
- `POST /admin-api/v1/gift-cards/batches`
- `GET /admin-api/v1/gift-cards/summary`
- `GET /admin-api/v1/gift-cards/cards`
- `POST /admin-api/v1/gift-cards/void`
- `GET /admin-api/v1/gift-cards/attempts`

用户侧：

- `POST /api/gift-cards/redeem`

用户侧兑换接口必须鉴权，并按 `user_id + IP` 做短期限流。默认 1 小时 10 次，可用：

- `GIFT_CARD_REDEEM_RATE_LIMIT_WINDOW_SECONDS`
- `GIFT_CARD_REDEEM_RATE_LIMIT_MAX_HITS`
- `GIFT_CARD_REDEEM_RATE_LIMIT_PRUNE_INTERVAL_SECONDS`

## 后续补齐

- 后台继续补批次详情钻取、批量发放对象、发放名单导入 / 导出和更细风控统计。
- 后台补批量发放名单和发放对象管理。
- 后台暂不做导出完整卡码；当前按“页面内查看和复制”满足自测和小规模发放。后续若做批量导出，必须加 owner / finance 权限、二次确认、导出原因和审计，并避免把文件落入仓库。
- 若后续礼品卡用于商务发放，需要增加发放对象、发放渠道、外部备注和批次归属。

## 风险和监控

需要重点监控：

- 同一用户 / IP 连续兑换失败。
- 同一礼品卡重复兑换：同一账号重复提交应表现为幂等成功且 App 提示“权益已生效”；其他账号重复兑换仍应失败。
- 兑换成功但权益发放失败。
- 已作废 / 已过期礼品卡被大量尝试。
- 礼品卡生成、导出和作废操作。

礼品卡接入支付权益、商务发放或更高风险批量操作后，必须同步更新用户协议、隐私政策、风险提示、项目记忆和本 runbook。
