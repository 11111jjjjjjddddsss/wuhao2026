# 礼品卡 Runbook

最后更新：2026-05-22

## 当前状态

礼品卡当前只是 Android 设置页里的占位入口，不是真实兑换系统。

当前真实行为：

- 设置页入口显示“礼品卡”三个字。
- 礼品卡页输入框不放提示词，前端不限制数字、英文或其他字符。
- 输入为空时“兑换”按钮禁用。
- 输入内容后点“兑换”或键盘 Done，只提示“礼品卡功能后续接入”。
- 当前不调用后端接口，不修改会员权益、每日额度、升级补偿、加油包、订单或 `quota_ledger`。
- debug-only 预览面板里的“兑换成功 / 确定”只是未来成功态样式预览，不代表真实兑换能力已接入。

## 当前没有的能力

- 没有后端礼品卡表。
- 没有用户侧兑换接口。
- 没有后台生成、发放、作废、查询或导出能力。
- 没有兑换记录、操作审计、撞库防护、失败限流。
- 没有真实权益发放事务。

## 接入原则

- 后端是礼品卡唯一真相来源。
- Android 只提交用户输入并展示后端结果，不在客户端判断卡是否有效，不在客户端直接发权益。
- 没有后端成功结果时，真实礼品卡页不能弹“兑换成功”卡片。
- 礼品卡规则必须先定清楚：卡类型、权益内容、有效期、可兑换次数、是否可叠加、是否绑定用户、是否可作废、异常处理和数据保留周期。

## 买服务器后建议最小表

后续真实接入时，至少需要两类数据：

- 礼品卡主表：卡号哈希、状态、权益类型、权益数量或会员档位、有效期、发放来源、创建时间、作废时间、操作人、备注。
- 兑换记录表：卡号哈希、用户 ID、兑换时间、兑换前后权益、幂等 key、失败原因、操作来源。

不要明文保存可直接撞库的完整卡号；如果业务必须显示卡号，也只在受控后台里展示脱敏值。

## 买服务器后建议接口

- `POST /api/gift-cards/redeem`
  - 用户侧兑换接口。
  - 必须鉴权。
  - 必须事务内校验卡状态并发权益。
  - 必须幂等，防止用户连点或弱网重试重复发权益。
  - 失败时返回明确但不过度泄露的原因，例如已兑换、已过期、不可用或格式无效。

## 管理后台最小能力

- 生成礼品卡。
- 发放 / 导出礼品卡。
- 查询卡状态。
- 作废礼品卡。
- 查看兑换记录。
- 查看失败次数和异常兑换。
- 所有操作记录操作人、时间、原因和变更前后值。

## 风险和监控

需要重点监控：

- 同一用户 / IP 连续兑换失败。
- 同一礼品卡重复兑换。
- 兑换成功但权益发放失败。
- 已作废 / 已过期礼品卡被大量尝试。
- 礼品卡生成或导出操作。

礼品卡接入支付、会员或加油包权益后，必须同步更新用户协议、隐私政策、风险提示、项目记忆和本 runbook。
