# 管理后台产品与页面设计

最后更新：2026-06-21

## 设计目标

后台是运营和排障工具，不是公开官网。第一版要让自己能快速回答三个问题：

- 现在服务是不是正常，哪里出错了。
- 某个用户为什么登录、问诊、上传、会员或反馈出了问题。
- 最近用户到底在卡什么，哪些功能和回答质量最值得改。

整体采用 Vite 静态前端 + Go 后端管理 API。界面风格克制、紧凑、清晰，优先信息密度和可操作性，不做营销页、插画页或大面积装饰。

当前说明：本文是后台产品和页面设计参考，不是唯一实现真相；已经上线的后台能力以 [management-backend.md](D:/wuhao/docs/runbooks/management-backend.md) 和当前代码为准。当前生产后台按“一个 owner 主账号自己用”的傻瓜工作台来做，页面优先清晰、直观、少解释；底层保留登录、审计和低权限巡检兜底，不把多人角色模型作为页面核心。本文中早期“建议 / 缺口 / 第一阶段顺序”的条目如果已经落地，要按当前状态理解，不再当作未实现结论。

## 总体信息架构

推荐左侧固定导航，按业务分组，不要把所有菜单平铺成一长串：

- 总览
- 用户与增长
  - 用户管理
  - 地区 / 来源统计
  - 用户聊天 / 反馈洞察
- 权益与交易
  - 会员 / 额度 / 加油包
  - 订单 / 订购
  - 礼品卡跟踪
- 运营工作台
  - 帮助与反馈
  - App 自动日志
  - 今日农情
  - 检查更新
- 安全与系统
  - 账号安全
  - 后台审计
  - 服务健康
  - 系统设置（后续）

第一版导航可以全部展示，但没有真实数据真源的模块只做“规划中 / 只读占位”，不要做假按钮。

正式上架检查区要区分“程序处理”和“人工确认”。人工确认项不是程序故障，适合展示 App 公安备案、AccessKey 轮换、短信套餐余额、SLS 首封告警邮件、最终 release 物料和最终真机回归等事项；每个这类条目应尽量给出“确认方式”，说明去哪里确认、确认后看什么证据，以及哪些敏感材料不能写进仓库、日志或后台备注。

顶部栏建议固定 48px 左右，放生产环境标识、全局搜索、时间范围、刷新按钮、账号安全入口和当前后台账号。生产环境要有明确但克制的“生产后台”标识，避免误操作。

## 当前可用数据和缺口

当前能直接利用的数据：

- 成功问诊轮次：`session_round_archive`、`session_round_ledger`。
- 问诊地区：`session_round_archive.region/source/reliability`、`session_ab.last_region*`。
- 额度消耗：`daily_usage`、`quota_ledger`。
- 会员 / 加油包 / 升级补偿：`user_entitlement`、`topup_packs`、`upgrade_credits`。
- 帮助与反馈：`support_messages` 和内部会话接口。
- App 自动日志：`client_app_logs`。
- 今日农情：`daily_agri_cards`。
- 后台操作：`admin_audit_logs`。
- API 请求 / 慢请求 / 5xx：Go 结构化日志和 SLS。

当前缺口：

- 缺 App 打开、页面曝光、入口点击、按钮点击等正向行为埋点。
- 缺普通 API 请求稳定携带 App 版本、系统、设备、地区字段；目前这些主要在失败日志里出现。
- 缺完整登录方式流水；当前新包只走普通短信验证码，历史 fusion 事件仅用于旧包排障。
- 缺上传元数据表，无法稳定区分 chat / support、成功 / 失败、文件大小和用途。
- App release 发布历史已接 `app_release_events`；当前仍缺 APK 上传、完整回滚入口和真机覆盖安装验收闭环。
- 缺今日农情 view / click / dismiss 行为。
- 后台总览 / 监控 / 产品洞察聚合 API 已接入首版；仍缺更细的登录精准漏斗、日报、人工标签和完整运营报表。

后续如果要把地区成交、版本漏斗和用户行为看准，需要补一个轻量行为事件表或事件日志接口；事件只记操作类型、时间、App 版本、设备、地区和脱敏用户，不记聊天正文、图片 URL、手机号或 token。

## 首页总览

首页是 1 屏到 1.5 屏内的运营驾驶舱。

推荐 12 栅格布局：

- 顶部一行 KPI 卡：服务健康、今日问诊、活跃 / 登录、错误事件、未回复反馈、今日农情。
- 中左 8 栅格：问诊量、图片上传、SSE 完成 / 失败、额度消耗趋势。
- 中右 4 栅格：待处理队列，未回复反馈、App error spike、今日农情失败、待发布版本。
- 下左：地区 / 来源排行，优先横向条形图，不做复杂地图。
- 下中：最近异常表，合并服务 5xx、App 自动日志、审计失败。
- 下右：今日农情预览、检查更新当前配置、关键资源状态。

顶部状态卡：

- 服务健康：API、RDS、Redis、OSS、SLS、模型 Key 池、DYPNS。
- 今日业务：活跃用户、问诊轮次、图片问诊轮次、帮助反馈数、App 日志错误数。
- 登录链路：短信验证码发送 / 登录成功失败、历史旧包融合事件、失败原因 Top。
- 模型链路：主聊天成功、SSE 中断、模型开流失败、联网搜索命中、Key failover。
- 成本与额度：今日扣次、Free / Plus / Pro 使用量、加油包消耗。

中部图表：

- 最近 24 小时请求量 / 5xx / 慢请求。
- App 错误事件 Top 10。
- 帮助反馈未处理队列。
- 省市分布热力表：注册用户、活跃用户、问诊用户、付费用户。

底部列表：

- 最新 5 条严重 App 日志。
- 最新 5 条待回复反馈。
- 最新 5 条后台审计。
- 今日农情生成状态。

- 会员档位
- 最近活跃时间
- 注册时间
- App 版本
- 设备型号
- 地区
- 地区来源：用户授权粗略定位 / IP / 未知
- 地区可信度：可靠 / 低可信

列表列：

- 用户
- 会员
- 今日剩余次数
- 加油包 / 升级补偿
- 最近问诊
- 最近登录
- 地区
- App 版本
- 最近错误
- 最近反馈

详情抽屉：

- 基本信息：账号、创建时间、最近活跃、登录方式、设备概况。
- 会员权益：档位、到期、每日额度、今日已用、加油包、升级补偿。

## 地区与来源

这个模块服务后续投放、销售和产品判断。

核心指标：

- 注册用户地区分布。
- 最近活跃用户地区分布。
- 问诊用户地区分布。
- 图片问诊用户地区分布。
- 会员成交用户地区分布。
- 加油包购买地区分布。
- 反馈 / 投诉地区分布。

数据口径：

- 优先使用用户授权粗略定位后的系统反查地区，标记为可靠。
- 未授权时使用 IP 粗定位，只做低可信参考。
- 支付接入后，成交地区不要只看支付 IP，还要结合最近可靠 App 地区。
- 地区统计默认按省 / 市两级聚合，不存经纬度，不做轨迹地图。

页面设计：

- 左侧为指标切换：注册、活跃、问诊、图片问诊、付费、反馈。
- 中间为省份排行和城市排行。
- 右侧为时间筛选、会员筛选、App 版本筛选。
- 点击地区进入用户列表，不展示个人敏感信息。

## 会员与额度

第一版先只读，等支付接入后再开放高风险操作。

需要展示：

- Free / Plus / Pro 用户数。
- Plus / Pro 到期分布。
- 每日额度使用分布。
- 今日次数耗尽用户。
- 加油包余额分布。
- 升级补偿次数余额。
- `quota_ledger` 扣次流水。

用户详情内可看：

- 今日额度记录。
- 加油包状态：active / used_up / expired。
- 升级补偿剩余。
- 最近扣次来源：daily / topup / upgrade。

高风险操作：

- 人工补偿次数。
- 手动修正会员。
- 作废加油包。

这些操作必须等后台账号、owner 权限、二次确认和审计都完成后再开放。

## 礼品卡

礼品卡需要单独做“批次 + 卡 + 兑换流水 + 风控”。

当前首版已经接入批次创建、卡状态列表、兑换尝试列表、Android 用户兑换和未兑换卡作废；新生成礼品卡完整卡码会加密保存，后台列表可直接查看和复制。旧版本生成且没有加密字段的卡只能展示掩码 / 尾号。

批次字段：

- 批次名
- 权益类型
- 数量
- 有效期
- 创建人
- 创建原因
- 状态：草稿 / 已生成 / 部分发放 / 已停用

单卡字段：

- 卡号 hash
- 完整卡码加密字段
- 脱敏展示码
- 所属批次
- 权益内容
- 状态：未发放 / 已发放 / 已兑换 / 已作废 / 已过期
- 发放对象
- 兑换用户
- 兑换时间

后台页面：

- 批次列表。
- 批次详情。
- 兑换记录。
- 失败 / 撞库 / 过期记录。
- 作废操作。

原则：

- 完整卡码只允许在受后台登录、角色权限和 CSRF 保护的礼品卡页查看 / 复制，不写进备注、审计 detail、日志、文档或聊天。
- 暂不做完整卡码批量导出；若后续做导出，必须受 owner / finance 权限、二次确认、原因记录和审计保护。
- 发放、作废、导出、兑换补偿全部写审计。

## 产品洞察

这是后台的长期核心。目标不是窥探用户，而是把真实问题聚合出来，让后续产品、提示词、UI 和后端修复有依据。

当前首版已接入：

- 后端 `GET /admin-api/v1/insights`。
- 前端“产品洞察”页。
- 今日 / 24h / 7d / 30d 聚合窗口：新增用户、登录 session、问诊轮次、去重问诊用户、图片问诊、额度消耗、App error / warn、登录排障、闪退补报、用户反馈、礼品卡兑换和今日农情失败。
- 质量信号：待回复反馈、反馈 open / replied / closed、检查更新启用和物料齐全状态、今日农情状态、可兑换礼品卡和 24h 失败兑换尝试。
- 固定分类：最近 30 天用户反馈主题关键词命中、最近 7 天 App 自动日志 event 分类、Top App 事件和礼品卡失败原因。
- 隐私边界：只返回计数、比例、事件名和固定分类，不返回聊天全文、反馈正文、图片 URL、手机号、token、模型 Key 或礼品卡完整码。

当前首版不是：

- 不是 AI 自动总结日报。
- 不是完整 `product_insight_reports` / `product_insight_items` 独立报表表。
- 不展示代表短摘、处理状态、责任人或严重度。
- 不把单条反馈正文、完整聊天原文、图片原图或敏感字段铺到后台。

数据来源：

- 帮助与反馈。
- App 自动日志。
- `session_round_archive` 最近 30 天问诊归档。
- 订单 / 礼品卡 / 支付异常。
- 后续用户行为埋点。

处理方式：

- 定期任务或手动任务读取最近 N 天数据。
- 先做脱敏和截断。
- 只抽取主题、问题类型、严重度、影响人数、代表性短摘和来源引用。
- 不把完整聊天正文、图片 URL、手机号、token 或模型 Key 放进洞察表。
- 结果写入 `product_insight_reports` / `product_insight_items` / `product_insight_sources` 这类聚合表。

洞察分类：

- 登录失败
- 验证码失败
- 图片上传失败
- 主聊天中断
- 回答不准 / 答偏
- 地区定位不准
- 追问太多 / 太少
- 会员权益疑问
- 检查更新失败
- 今日农情问题
- UI 看不懂
- 新功能诉求

页面展示：

- 本周最该修 Top 10。
- 影响用户数。
- 严重度。
- 代表短摘。
- 证据来源数量。
- 建议改动。
- 处理状态。
- 责任模块：Android / Go 后端 / 提示词 / 云资源 / 运营。

后续 Codex 可以优先读取这些脱敏洞察报表，而不是直接扫生产库完整聊天。

## 审计日志

所有后台关键动作都能查：

- 登录 / 登出
- 登录失败
- 查询用户
- 查看敏感详情
- 回复反馈
- 修改版本
- 手动补跑今日农情
- 补权益
- 礼品卡生成 / 作废 / 导出
- 订单补偿
- 修改后台账号权限

审计列表默认按时间倒序，支持 actor、action、target_user、success、时间筛选。

## 后台账号边界

当前生产只有一个 owner 主账号日常使用，界面按主账号后台设计，避免把“权限 / 角色”放成普通使用者需要理解的概念。底层保留最小服务端授权字段和低权限巡检账号，用来跑自动 smoke、降低主账号密码扩散风险，并给未来多人运营留口子。

后续如果真的多人协作，再把客服、内容、发布、财务、审计等分工重新产品化；当前不要为了“看起来规范”在页面上堆复杂角色说明。

## 账号初始化

后台账号不能写进仓库，也不能把明文密码放进前端。

推荐方式：

1. 后端支持一次性 bootstrap 环境变量或 Cloud Assistant 初始化脚本。
2. 初始化时把密码 hash 后写入 `admin_users`。
3. 初始化成功后删除环境变量或禁用 bootstrap。
4. 当前已有账号安全页和强制改密硬兜底，多账号运营前继续补受控重置、账号禁用和角色管理流程。
5. 生产密码必须足够长且唯一，不建议使用简单重复口令。

## 页面视觉规范

- 背景：浅灰工作台，不用大渐变。
- 侧栏：深色或白底窄侧栏均可，优先稳定扫描。
- 卡片：只用于指标和列表容器，圆角不超过 8px。
- 表格：密度适中，行高稳定，支持筛选、分页、排序。
- 详情：优先右侧抽屉，避免频繁跳页。
- 颜色：绿色只作为品牌强调，不铺满整页。
- 状态色：成功绿、警告橙、错误红、信息蓝，保持克制。
- 操作按钮：危险操作必须二次确认，不能和普通按钮同样视觉重量。
