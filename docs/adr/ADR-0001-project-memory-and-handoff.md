# ADR-0001：项目长期记忆与跨窗口交接机制

- 状态：已采纳
- 日期：2026-04-14

## 背景

项目将长期迭代 Android 前端、Go 后端以及后续运维能力。单纯依赖聊天历史、人工口述背景或模型自带记忆，会导致换窗口后重新对齐成本高、旧方案容易复活、运维口径散落。

## 决策

采用“仓库内结构化记忆”作为长期交接机制：

- 根 `AGENTS.md` 作为唯一主规则
- `app/AGENTS.md`、`server-go/AGENTS.md` 作为局部执行补充
- `docs/project-state` 维护当前状态、未关闭风险、待决策、近期重要变更
- `docs/adr` 维护关键方案取舍与废弃原因
- `docs/runbooks` 维护运维与排障入口

所有 Codex 在开始工作时默认读取以上文件；在完成任务时，若当前真相、风险、待决策、运维入口或方案取舍发生变化，必须同步回写。

## 为什么这么定

- 避免把关键知识锁在单个聊天窗口里
- 降低模型幻觉，因为记忆以仓库真实文件为准
- 让“为什么这么改”有固定沉淀位置，减少旧方案回流
- 为后续运维、发版、查日志、查库等能力预留统一入口

## 影响

正面影响：

- 新窗口接手成本更低
- 同一问题更容易追到当前真相和历史决策
- 后续可以渐进式补运维脚本和 GitHub 协作层

负面影响：

- 需要 Codex 在每次任务后自觉维护这些文件
- 若后续长期漏更，文档仍可能漂移

## 后续约束

- 过期方案不能与当前方案长期并列，保留时必须明确标记为历史归档或已废弃
- 关闭的风险和已决事项要及时迁出当前清单
- 后续若引入自动化校验，再单独补一条 ADR
