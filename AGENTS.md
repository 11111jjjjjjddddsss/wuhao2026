# 农技千问仓库规则

## Chat UI 铁规则

- 聊天框的 UI 行为必须由代码直接决定，不能依赖本地缓存、旧历史、旧底部视口、旧流式草稿才能正常。
- 允许清除数据后丢失历史、底部视口、流式草稿和本地 `user_id`；不允许因此把聊天框 UI 打回旧毛病。
- 任何聊天 UI 相关改动，如果在 clean-state 下才暴露问题，也算真实 bug，不能按“用户清数据后特殊情况”忽略。

## Clean-State 定义

- 无本地聊天记录
- 无本地底部视口
- 无本地流式草稿
- 全新 `user_id`

对 Android 来说，这等价于：

- 清除应用数据后首次启动

## 必须执行的回归检查

只要改到下面任一范围，就必须执行 clean-state 回归检查：

- 聊天列表布局
- 发送后上抬 / 小球锚点
- 流式生成渲染
- 自动滚动 / 手动打断 / 回到底部按钮
- 底部输入区、免责声明、底部白板、遮挡关系
- 冷启动 / 重进 app / 历史恢复

## 验收标准

- 清除数据后首次进入，聊天页不能白屏、透明、空白占位异常。
- 清除数据后首次发送，用户气泡上抬、小球锚点、生成区与输入区避让必须正常。
- 生成区正文、免责声明、最后一段内容不能被输入框或底部区域遮住。
- 生成中手动上滑、下滑不能被错误抢手；只有真正回到底部后，自动跟随才能重新接管。
- 生成过程中不能出现明显的“飞字、冒头、跳字、整段突然蹦出、先紧后松再重排”。
- 重开 app 时，若本地有历史，则恢复逻辑必须稳定；若本地无历史，也不能因此触发旧 UI 毛病。

## 开发约束

- 不要把锚点高度、渲染模式、底部避让、滚动规则做成本地设置项。
- 本地缓存只能用于恢复数据，不得用于决定聊天框是否“看起来正常”。
- 如果一个修复只有在“已有历史 / 已有视口缓存”时才成立，该修复视为不合格。
- 提交聊天 UI 改动时，默认同时检查 clean-state 和有历史两种状态。

## Backend 约束

- 当前生产主后端以 [server](D:/wuhao/server) 为准。
- 图片上传能力默认并入主后端，不单独拆服务，除非后期确有独立扩容或独立安全隔离需求。
- 商城能力后续优先在 [server](D:/wuhao/server) 内按模块扩展，先预留清晰路由和数据边界，不提前搭空框架或第二套服务。
- 后端边界和后续商城预留方式见 [docs/backend-boundaries.md](D:/wuhao/docs/backend-boundaries.md)。

## 运维接入规则

- 后期如果接入阿里云 SAE、PolarDB、SLS，默认允许 Codex 辅助运维，但前提是工作区或机器上已经配置好可控接入方式，不能假设 Codex 天然拥有云端权限。
- SAE 相关操作优先走可审计的脚本、CLI 或 OpenAPI，不优先依赖控制台手工点击路径。
- PolarDB 相关操作默认区分两类：
  - 人工日常查看优先走 DMS 或控制台可视化入口。
  - Codex 诊断、查询、发布辅助优先走只读数据库账号、迁移脚本、备份脚本、巡检脚本，不默认直接裸改生产库。
- 生产运维默认遵守最小权限原则：能只读就不写，能灰度就不直改，能走脚本就不走临场手工。
- 如果未来要让 Codex 长期帮忙发版、回滚、查日志、查库，优先在仓库内固化这些入口：
  - 部署脚本
  - 回滚脚本
  - SAE 日志查询脚本
  - PolarDB 只读查询或备份脚本
- 后续任何 GPT 读到本仓库时，都应默认把“可脚本化、可审计、可回滚”作为 SAE / PolarDB 运维第一原则，而不是把生产控制台手工操作当成主路径。
